## Claim 8 — Does the OpenChoreo client expose SecretReference / GitSecret operations today?

Searched `asdlc-service/clients/openchoreo/` (the only OC client; git-service has no OC API client — it talks to k8s directly via controller-runtime).

Two layers:

1. **Generated client (`asdlc-service/clients/openchoreo/gen/client.gen.go` + `gen/types.gen.go`) — FULLY EXPOSES both.**
   - `SecretReference` (full CRUD): `ListSecretReferences`, `CreateSecretReference(+WithBody)`, `GetSecretReference`, `UpdateSecretReference(+WithBody)`, `DeleteSecretReference` — interface decls at client.gen.go:532-549; impls at 2619-2692; types `SecretReference`/`SecretReferenceSpec`/`SecretReferenceStatus`/`SecretReferenceList` at types.gen.go:3355-3395; `WithResponse` variants also generated.
   - `GitSecret` (List/Create/Delete only — NO Get, NO Update): `ListGitSecrets`, `CreateGitSecret(+WithBody)`, `DeleteGitSecret` — interface decls at client.gen.go:663-672; impls at 3195-3231; request type `CreateGitSecretRequest` with `SecretType` enum `basic-auth`/`ssh-auth` and `WorkflowPlaneKind` enum at types.gen.go:2143-2174; `GitSecretResponse`/`GitSecretListResponse` at 2551-2567.

2. **Hand-written wrapper clients — DO NOT wrap any secret operation.**
   - Only `project_client.go`, `component_client.go`, `namespace_client.go` exist (plus `helpers.go`, `transport.go`). None wrap SecretReference or GitSecret. `component_client.go` only *references* `SecretRef`/`AnthropicSecretRef` as plain string parameters passed into component build params (component_client.go:132-138, 770-771) — it does not call any OC secret API.

**Comparison to AM (`/Users/wso2/repos/agent-manager/agent-manager-service/clients/openchoreosvc/client/`):**
- AM `secret_references.go`: `CreateSecretReference`, `GetSecretReference`, `ListSecretReferences`, `UpdateSecretReference`, `DeleteSecretReference`, plus helper `GetWorkloadSecretRefNames`.
- AM `git_secrets.go`: `CreateGitSecret`, `ListGitSecrets`, `DeleteGitSecret` (no Get/Update — same surface as the generated layer).
- AM has hand-written, typed wrappers (returning `*SecretReferenceInfo`/`*GitSecretInfo`) over its generated layer. labs-agentic-engineer has the *generated* layer with identical operation coverage but is **missing the hand-written wrapper layer** for secrets.

**Feasibility answer:** A refactor that emits SecretReference / GitSecret does NOT need client regeneration. The generated client already supports SecretReference (full CRUD) and GitSecret (Create/List/Delete) against a `NamespaceNameParam`. What is missing is the thin hand-written wrapper + mocks (analogous to AM's `secret_references.go` / `git_secrets.go`) to expose these through the `asdlc-service/clients/openchoreo` surface that the rest of the platform consumes. Caveat: GitSecret has no Get/Update in either the generated client or AM, so any flow needing read-back/update of a GitSecret must use List, or that op must be added to the OpenAPI spec and regenerated.
